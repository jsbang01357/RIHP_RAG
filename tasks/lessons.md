# 작업 교훈

## PDF 추출

- 텍스트층이 있어도 큰 제목 글자가 세 번씩 중복될 수 있으므로 별도 탐지한다.
- 표는 일반 텍스트 추출만으로 병합 셀과 열 관계가 손실될 수 있어 전수 검수한다.
- 계간지는 호 전체보다 기사 단위가 사람과 RAG 모두에 적합하다.
- 보고서는 물리 PDF 페이지와 인쇄 페이지가 다를 수 있으므로 물리 페이지를 기준으로 남긴다.
- `.pdf.download` 안의 PDF는 완전한 파일로 가정하지 않고 파서 검증을 먼저 한다.

## 저장소

- 한글 제목은 frontmatter에 보존하고 파일명은 ASCII ID를 사용한다.
- 원본 PDF와 생성 콘텐츠를 같은 Git 이력에 넣지 않는다.
- Markdown을 기준 원본으로 두고 임베딩·벡터 인덱스는 재생성 산출물로 취급한다.

## 동기화

- 수집 한도 `0`은 명시적인 비활성화로 처리하고, 네트워크 실행 전에 단위 테스트한다.
- 메타데이터만 고칠 때도 수집 선택 로직이 실행될 수 있으므로 `--dry-run` 결과를 먼저 확인한다.

## 원문 연결

- RIHP는 외부 `Referer` 또는 무출처 게시판 상세 요청을 홈페이지로 `302`한다.
- 화면에는 세션 의존적인 직접 PDF 링크를 노출하지 않고 공식 게시판·글번호를 남긴다.
- RIHP 링크 검증은 내부 `Referer`로 게시물 존재를 확인하는 검사와 외부 브라우저 동작 검사를 구분한다.

## 검색

- 모든 입력 단어의 완전 일치를 요구하면 추천 키워드는 동작해도 자연스러운 질문은 쉽게 0건이 된다.
- 한글 조사를 제거하고 불용 질문어를 제외한 뒤 핵심 개념 두 개 이상을 점수화해 상위 결과를 정렬한다.
- 한글 IME 조합 중에는 입력값을 다시 쓰지 말고 `compositionend` 이후 검색 상태를 갱신한다.
- 청크를 그대로 카드로 표시하면 같은 발간물과 PDF 페이지가 반복되어 홈페이지 제목 검색과 차이가 흐려진다.
- 검색 청크는 `발간물(source_id) → PDF 페이지 → 최고 점수 청크` 순으로 묶고, 같은 본문 지문도 한 번만 표시한다.
- 첫 화면의 설명과 결과 구조에서 PDF 본문 검색, 발간물 묶음, 페이지 근거를 함께 드러내야 탐색 도구의 차별점이 보인다.

## UI

- 검색·자료 현황·다운로드·안내를 한 화면에 길게 쌓으면 핵심 작업이 흐려지므로 데스크톱은 고정 사이드바, 모바일은 상단 탭으로 작업 공간을 분리한다.
- 화면을 분리해도 검색어와 필터 상태는 유지하고 URL의 `view` 매개변수와 브라우저 뒤로가기를 함께 동기화한다.

## Haystack·GCP

- 문서 임베딩은 빌드 시 한 번 생성해 컨테이너에 포함하고, 요청 시에는 질문 임베딩만 생성하면 비용과 지연을 줄일 수 있다.
- 정적 인덱스에서 중복을 제거했더라도 하이브리드 검색 결과는 `발간물 + PDF 페이지` 기준으로 다시 중복 제거한다.
- 생성 모델 장애가 검색 전체 장애가 되지 않도록, 근거 카드와 RIHP 링크를 반환하는 검색 전용 폴백을 유지한다.
- 생성 프롬프트는 근거 밖 수치 환산·추정을 금지하고 제공된 근거 번호만 사용하도록 명시한다.
- 커스텀 도메인은 새 Cloud Run 주소와 API를 먼저 검증한 뒤 DNS를 바꾸고, 기존 Pages 주소는 비상용으로 남긴다.
